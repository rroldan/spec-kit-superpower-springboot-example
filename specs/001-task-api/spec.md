# Task API

## Short name

task-api

## Overview

Create a RESTful API for managing tasks. This feature covers the Task entity (id, title, description, status, createdAt, updatedAt) and provides a first-iteration endpoint to create tasks: `POST /tasks`.

## Actors

- **User**: any authenticated client that creates and manages tasks
- **System**: validates input, persists tasks, and returns appropriate HTTP responses

## User Scenarios

1. A user submits a request to create a new task with title and optional description. The system validates input, creates the task with status `TODO`, and returns `201 Created` with the created representation and `Location` header.
2. A user submits an invalid request (missing or empty title, or invalid status). The system returns `400 Bad Request` with structured validation errors.

## Functional Requirements

1. `POST /tasks` creates a new task when the request is valid.
   - Request body must be JSON and validated.
   - On success, response `201 Created` includes `Location` header pointing to `/tasks/{id}` and a body with the created task.
   - The created task contains `id` (UUID), `title` (string), `description` (string, optional), `status` (enum: TODO, IN_PROGRESS, DONE), `createdAt` (ISO-8601 instant), `updatedAt` (ISO-8601 instant).
2. Input validation failures return `400 Bad Request` with structured error payload containing field-level messages.

## Acceptance Criteria (testable)

- Given valid `title` (non-empty) and optional `description`, `POST /tasks` returns `201 Created`, response body contains the task with `status=TODO`, and `Location` header points to the new task URI.
- Given missing or blank `title`, `POST /tasks` returns `400 Bad Request` with an error describing the `title` field requirement.
- The `id` returned is a UUID and `createdAt`/`updatedAt` are present and parsable as ISO-8601 instants.

## Success Criteria

- 95% of valid create requests return within 500ms under normal load (measured in integration tests).
- 100% of invalid requests return structured validation errors with HTTP `400`.

## Key Entities

- Task
  - `id: UUID` (generated by DB)
  - `title: String` (required, non-empty, max length 255)
  - `description: String` (optional, max length 4000)
  - `status: Enum` (TODO, IN_PROGRESS, DONE) — default `TODO` on create
  - `createdAt: Instant` (generated)
  - `updatedAt: Instant` (generated)

## API: Create Task

- Endpoint: `POST /api/v1/tasks`
- Request: `Content-Type: application/json`

Request schema (example):

```json
{
  "title": "Buy groceries",
  "description": "Milk, bread, eggs"
}
```

- Responses:
  - `201 Created` — body: created Task representation, header `Location: /api/v1/tasks/{id}`
  - `400 Bad Request` — body: `ErrorResponse` with validation details

## Validation Rules

- `title`: required, non-blank, max 255 characters
- `description`: optional, max 4000 characters
- `status`: must not be supplied by clients on creation (server sets `TODO`). If supplied, request is rejected with `400`.

## Assumptions

- Authentication is required for all endpoints (per project constitution). This spec focuses on the authenticated happy path; auth details are handled by platform-level configuration.
- Database generates `UUID` primary keys (gen_random_uuid()).
- Timestamps are generated by the persistence layer and returned in responses.

## Non-functional

- API is versioned under `/api/v1`.
- Uses JSON and follows standard REST semantics.
